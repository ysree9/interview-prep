What is Spring Framework and why is it used?

30-second answer

Spring Framework is a Java application framework that provides dependency injection, bean management, AOP, transaction support, MVC, data access integration, and more. It is used to build modular, testable, loosely coupled applications.

Deep dive

Spring Framework is a Java framework whose core idea is Inversion of Control (IoC). Instead of your classes creating all of their own dependencies, Spring creates and manages those objects for you as beans.

How it works
  • Spring scans configuration metadata such as @Component, @Service, @Repository, @Configuration, and @Bean.
  • It registers BeanDefinitions that describe how each bean should be created.
  • The IoC container creates beans, resolves dependencies, injects them, and manages their lifecycle.
  • Spring also adds infrastructure features such as transactions, AOP, MVC, security integration, data access support, and event handling.
Example / mental model
Without Spring:
OrderService service = new OrderService(new PaymentService());

With Spring:
@Service
class OrderService {
    private final PaymentService paymentService;

    OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}

Spring creates both objects and injects PaymentService into OrderService.
Common pitfalls
  • Spring does not magically remove complexity; it centralizes object creation and infrastructure concerns.
  • The real benefit is loose coupling, testability, lifecycle management, and framework integration.
Remember: Spring Framework = IoC container + enterprise infrastructure around your application objects.

What is Dependency Injection (DI)?

30-second answer

Dependency Injection means an object receives the dependencies it needs from an external container instead of creating them itself. In Spring, the IoC container creates beans, resolves dependencies, and injects them, which reduces coupling and improves testability.

Deep dive

Dependency Injection means an object receives the dependencies it needs from outside instead of constructing them itself. Spring uses DI as the main mechanism for implementing IoC.

How it works
  • Spring discovers or registers the required beans.
  • When creating a bean, Spring inspects its constructor, fields, or setters for dependencies.
  • It resolves a matching bean, usually by type.
  • If multiple candidates exist, @Qualifier or @Primary can help choose one.
  • Spring injects the selected dependency into the target bean.
Example / mental model
@Service
class CustomerService {
    private final OrderService orderService;

    CustomerService(OrderService orderService) {
        this.orderService = orderService;
    }
}

Spring resolves OrderService and effectively does:
new CustomerService(orderServiceBean);
Common pitfalls
  • Constructor injection is preferred for required dependencies.
  • Field injection hides dependencies and makes testing harder.
  • Setter injection is useful when a dependency is genuinely optional.
Remember: DI answers: which dependency should this object receive, and how does it get there?
Likely interview follow-up Constructor injection vs setter injection vs field injection.

Difference between @Component, @Service, and @Repository.

30-second answer

@Component is the generic stereotype. @Service is intended for service-layer classes, and @Repository is intended for persistence-layer classes and also enables persistence exception translation. All three are discovered by component scanning.

Deep dive

@Component, @Service, and @Repository are stereotype annotations used by component scanning to register classes as Spring beans.

How it works
  • @Component is the generic stereotype.
  • @Service is intended for business/service-layer classes.
  • @Repository is intended for persistence/data-access classes.
  • @Repository also participates in persistence exception translation, converting certain vendor-specific persistence exceptions into Spring's DataAccessException hierarchy.
Example / mental model
@Component
class UtilityComponent {}

@Service
class PaymentService {}

@Repository
class CustomerRepository {}
Common pitfalls
  • @Service is mostly semantic; it communicates architectural intent.
  • @Repository has extra persistence-related behavior beyond naming.
  • All three ultimately result in bean registration.
Remember: Use the most specific stereotype because it improves readability and communicates layer responsibility.
Likely interview follow-up Does @Service behave differently from @Component?

What are Spring bean scopes?

30-second answer

Common Spring scopes are singleton, prototype, request, session, application, and websocket. Singleton is the default and gives one bean per ApplicationContext. Prototype creates a new instance each time the bean is requested.

Deep dive

A bean scope defines how long a bean instance lives and how many instances Spring creates.

How it works
  • singleton: one bean instance per ApplicationContext; this is the default.
  • prototype: a new instance every time Spring is asked for the bean.
  • request: one instance per HTTP request.
  • session: one instance per HTTP session.
  • application: one instance per ServletContext.
  • websocket: one instance per WebSocket session.
Example / mental model
@Scope("prototype")
@Component
class ReportBuilder {}

Each getBean(ReportBuilder.class) call creates a new ReportBuilder.
Common pitfalls
  • A Spring singleton is not automatically thread-safe.
  • Injecting a prototype directly into a singleton does not create a fresh prototype on every method call.
Remember: Scope controls object lifetime and instance reuse.

Difference between ApplicationContext and BeanFactory.

30-second answer

BeanFactory is the basic IoC container and provides lazy bean creation and dependency injection. ApplicationContext extends BeanFactory and adds features such as events, internationalization, resource loading, environment support, and automatic BeanPostProcessor registration.

Deep dive

BeanFactory is the basic Spring IoC container. ApplicationContext builds on BeanFactory and adds application-level features.

How it works
  • BeanFactory handles bean creation, dependency resolution, and lifecycle basics.
  • ApplicationContext extends BeanFactory.
  • ApplicationContext adds event publishing, internationalization, resource loading, environment/property support, and automatic registration of post-processors.
  • Most Spring applications interact with ApplicationContext rather than raw BeanFactory.
Example / mental model
ApplicationContext context =
    new AnnotationConfigApplicationContext(AppConfig.class);

MyService service = context.getBean(MyService.class);
Common pitfalls
  • BeanFactory is lower-level infrastructure.
  • ApplicationContext does not replace BeanFactory; it builds on it.
Remember: BeanFactory is the core container; ApplicationContext is the richer application container.

How does @Autowired work?

30-second answer

@Autowired asks Spring to resolve and inject a matching bean, primarily by type. If multiple candidates exist, Spring may need @Qualifier or @Primary. Constructor injection is generally preferred because required dependencies are explicit and the object can be immutable.

Deep dive

@Autowired tells Spring to resolve and inject a dependency. Resolution is primarily type-based.

How it works
  • Spring inspects the injection point.
  • It asks the BeanFactory for beans assignable to that type.
  • If exactly one candidate exists, Spring injects it.
  • If multiple candidates exist, Spring considers @Qualifier, @Primary, and other metadata.
  • If none exists and the dependency is required, bean creation fails.
Example / mental model
@Service
class CheckoutService {
    private final PaymentService paymentService;

    @Autowired
    CheckoutService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}

With a single constructor, @Autowired is usually optional in modern Spring.
Common pitfalls
  • Multiple matching beans can cause NoUniqueBeanDefinitionException.
  • Autowired field injection works, but constructor injection is generally cleaner.
Remember: @Autowired is dependency resolution + injection, not object creation by itself.
Likely interview follow-up How does Spring choose between multiple beans of the same type?

What is @Qualifier in Spring?

30-second answer

@Qualifier narrows bean selection when multiple beans match the same type. You place it at the injection point, and optionally on the bean definition, to tell Spring exactly which candidate to inject.

Deep dive

@Qualifier helps Spring choose the right bean when multiple beans match the same type.

How it works
  • Spring first finds type-compatible candidates.
  • @Qualifier narrows that candidate set.
  • The qualifier can be a string name or custom qualifier annotation.
  • @Qualifier is especially useful when several implementations of one interface exist.
Example / mental model
interface PaymentService {}

@Service("stripe")
class StripePaymentService implements PaymentService {}

@Service("paypal")
class PaypalPaymentService implements PaymentService {}

@Autowired
@Qualifier("stripe")
private PaymentService paymentService;
Common pitfalls
  • Relying only on variable names for disambiguation is less explicit.
  • @Qualifier is usually more precise than @Primary.
Remember: @Qualifier says: inject this specific candidate.

What is the purpose of @Configuration?

30-second answer

@Configuration marks a class as a source of bean definitions. Methods annotated with @Bean inside it can create and configure objects that Spring registers and manages in the ApplicationContext.

Deep dive

@Configuration marks a class as a source of bean definitions.

How it works
  • Spring processes the class during configuration parsing.
  • Methods annotated with @Bean contribute BeanDefinitions.
  • Those returned objects become Spring-managed beans.
  • Full configuration classes can be enhanced so calls between @Bean methods preserve container semantics.
Example / mental model
@Configuration
class AppConfig {

    @Bean
    PaymentService paymentService() {
        return new PaymentService();
    }

    @Bean
    OrderService orderService() {
        return new OrderService(paymentService());
    }
}
Common pitfalls
  • @Configuration is not the same as @Bean: one marks the class, the other marks bean factory methods.
  • Using @Bean is especially useful for third-party classes you cannot annotate.
Remember: @Configuration defines where bean configuration lives; @Bean defines individual beans.

Explain Spring bean lifecycle?

30-second answer

Spring creates the bean, injects dependencies, invokes aware callbacks, BeanPostProcessors before initialization, lifecycle callbacks such as @PostConstruct or InitializingBean, then BeanPostProcessors after initialization. On shutdown, destruction callbacks such as @PreDestroy run for eligible beans.

Deep dive

Spring bean lifecycle is the sequence from bean definition to creation, dependency injection, initialization, use, and destruction.

How it works
  • 1. Spring reads a BeanDefinition.
  • 2. The bean instance is created.
  • 3. Dependencies/properties are populated.
  • 4. Aware callbacks may run.
  • 5. BeanPostProcessor before-initialization callbacks run.
  • 6. @PostConstruct / InitializingBean / custom init method runs.
  • 7. BeanPostProcessor after-initialization callbacks run; a proxy may be returned here.
  • 8. The bean is ready for normal use.
  • 9. On context shutdown, @PreDestroy / DisposableBean / custom destroy methods may run.
Example / mental model
@PostConstruct
public void init() {
    // initialization work
}

@PreDestroy
public void cleanup() {
    // cleanup work
}
Common pitfalls
  • Instantiation and initialization are different stages.
  • Prototype beans do not get the same full destruction management as singletons.
Remember: Spring manages much more than just new Object(); it manages the whole bean lifecycle.
Likely interview follow-up BeanFactoryPostProcessor vs BeanPostProcessor.

30-second answer

XML configuration defines beans and dependencies in external XML files. Annotation or Java configuration uses @Component scanning, @Configuration, and @Bean. Modern Spring applications usually prefer annotation/Java configuration because it is type-safe and closer to the code.

Spring supports multiple ways to define beans. XML, annotations, and Java configuration all ultimately produce BeanDefinitions in the container.

  • XML uses <bean> definitions and explicit wiring.
  • Component scanning uses annotations such as @Component, @Service, and @Repository.
  • Java configuration uses @Configuration and @Bean.
  • Modern Spring projects usually prefer annotations and Java config because they are closer to the code and more type-safe.
Example / mental model

Java:
@Bean
PaymentService paymentService() {
    return new PaymentService();
}
Common pitfalls
  • XML is still valid and can coexist with Java configuration.
  • The container ultimately works with BeanDefinitions regardless of configuration style.
Remember: Different configuration styles, same underlying IoC container.

Explain AOP (Aspect-Oriented Programming).

30-second answer

AOP separates cross-cutting concerns such as logging, security, transactions, and metrics from business logic. Spring typically implements AOP using proxies that intercept method calls and apply advice around matching join points.

Deep dive

AOP separates cross-cutting concerns from business logic. Instead of repeating logging, transactions, security, or metrics in every method, Spring can apply that behavior through proxies.

How it works
  • Aspect: class containing cross-cutting logic.
  • Advice: what should run, such as before/after/around.
  • Pointcut: which method calls should be intercepted.
  • Join point: the intercepted method execution.
  • Proxy: wrapper that intercepts the method call and invokes advice around the target.
Example / mental model
@Aspect
@Component
class LoggingAspect {

    @Around("execution(* com.example.service..*(..))")
    Object log(ProceedingJoinPoint pjp) throws Throwable {
        System.out.println("Before");
        Object result = pjp.proceed();
        System.out.println("After");
        return result;
    }
}
Common pitfalls
  • Spring AOP is proxy-based.
  • Self-invocation inside the same object can bypass the proxy.
  • Final/private methods can limit proxy interception depending on proxy strategy.
Remember: AOP = intercept method calls and apply reusable cross-cutting behavior.
Likely interview follow-up Why does self-invocation break @Transactional?

How does @Transactional work?

30-second answer

@Transactional is usually implemented through a Spring proxy. A call entering the proxy starts or joins a transaction through a PlatformTransactionManager, invokes the target method, then commits on success or rolls back based on configured exception rules. Self-invocation can bypass the proxy.

Deep dive

@Transactional is usually implemented using Spring AOP proxies.

How it works
  • An external caller invokes the Spring proxy.
  • The proxy's transaction interceptor reads transaction metadata.
  • PlatformTransactionManager begins or joins a transaction.
  • The target business method runs.
  • On success, Spring commits.
  • On matching failure, Spring rolls back.
  • Propagation decides whether to reuse or create a transaction; isolation controls database visibility behavior.
Example / mental model
@Transactional
public void placeOrder() {
    orderRepository.save(order);
    paymentRepository.save(payment);
}

If the method throws a RuntimeException, the transaction normally rolls back.
Common pitfalls
  • Self-invocation can bypass the transactional proxy.
  • Private methods are not a good target for proxy-based @Transactional behavior.
  • Checked exceptions do not trigger rollback by default unless configured.
Remember: @Transactional is proxy interception around a method boundary.
Likely interview follow-up REQUIRED vs REQUIRES_NEW, and rollback rules.

What are proxy objects in Spring?

30-second answer

A Spring proxy is a wrapper around a target bean that intercepts method calls to apply cross-cutting behavior such as transactions, security, caching, or AOP advice. Spring commonly uses JDK dynamic proxies for interfaces and CGLIB subclass proxies for classes.

Deep dive

A Spring proxy is an object that stands in front of the real target bean and intercepts eligible method calls.

How it works
  • The caller talks to the proxy, not directly to the raw target.
  • The proxy runs advice such as transactions, security, caching, or metrics.
  • The proxy then delegates to the target method.
  • Spring may use JDK dynamic proxies or subclass-based proxies.
Example / mental model
Controller
   ↓
Service Proxy
   ↓
Transaction Interceptor
   ↓
Actual Service Bean
   ↓
Repository
Common pitfalls
  • Internal calls like this.someMethod() do not leave the object and re-enter through the proxy.
  • That is why some annotations appear to 'not work' on self-invoked methods.
Remember: Proxy = interception layer around your bean.

How do you implement custom annotations in Spring?

30-second answer

Create a Java annotation with @Target and @Retention, then give it meaning through AOP, a BeanPostProcessor, validation, or another Spring extension point. A common pattern is a custom annotation plus an @Aspect that intercepts annotated methods.

Deep dive

A custom annotation is just metadata until some framework code interprets it.

How it works
  • Create the annotation with @Target and @Retention(RUNTIME).
  • Apply it to methods/classes.
  • Use an @Aspect, BeanPostProcessor, validation framework, or other extension point to detect it.
  • Execute the custom behavior when the annotation is found.
Example / mental model
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface TrackExecutionTime {}

@Around("@annotation(TrackExecutionTime)")
public Object track(ProceedingJoinPoint pjp) throws Throwable {
    long start = System.currentTimeMillis();
    try {
        return pjp.proceed();
    } finally {
        System.out.println(System.currentTimeMillis() - start);
    }
}
Common pitfalls
  • Annotation alone does nothing.
  • Choose the extension mechanism based on whether you need method interception, validation, bean customization, etc.
Remember: Custom annotation + interpreter = reusable domain behavior.

How does Spring manage circular dependencies?

30-second answer

A circular dependency occurs when bean A depends on B and B depends on A. Spring can resolve some singleton setter/field cycles using early references, but constructor cycles cannot be resolved. The best fix is usually to redesign responsibilities and break the cycle.

Deep dive

A circular dependency means two beans depend on each other. Spring can resolve some singleton property-injection cycles using early references, but not constructor cycles.

How it works
  • Spring starts creating bean A.
  • A is instantiated but not fully initialized.
  • Spring exposes an early reference factory.
  • A needs B, so Spring starts creating B.
  • B needs A, so Spring can obtain A's early reference.
  • B finishes creation.
  • A receives B and finishes creation.
  • Eventually the final singleton reference is stored.
Example / mental model
A -> B
B -> A

Setter injection may sometimes be resolved.

Constructor injection:
new A(new B(new A(...)))
has no valid starting point.
Common pitfalls
  • Circular dependencies often signal poor design.
  • Early references become more complex when proxies are involved.
  • Constructor cycles cannot be resolved because neither object can be instantiated first.
Remember: Spring can sometimes break a singleton setter cycle, but redesign is usually better.
Likely interview follow-up singletonObjects vs earlySingletonObjects vs singletonFactories.

What is the difference between Singleton and Prototype beans?

30-second answer

A singleton bean has one instance per Spring ApplicationContext. A prototype bean creates a new instance every time Spring is asked for it. Spring manages the full lifecycle of singletons, but for prototypes it mainly handles creation and initialization.

Deep dive

Singleton and prototype differ in how many instances Spring creates and how long it manages them.

How it works
  • Singleton: one instance cached per ApplicationContext.
  • Prototype: new instance each time requested from the container.
  • Singletons are usually created once and fully lifecycle-managed.
  • Prototype beans are created and initialized, but Spring does not automatically manage their full destruction lifecycle.
Example / mental model

Common pitfalls
  • Prototype injected into singleton is resolved once unless you use ObjectProvider/@Lookup.
  • Prototype is not a substitute for thread safety.
Remember: Singleton = reused instance; prototype = new instance per lookup.

Explain constructor injection vs setter injection.

30-second answer

Constructor injection is best for required dependencies because it makes them explicit, supports immutability, and simplifies testing. Setter injection is useful for optional or reconfigurable dependencies. Field injection is generally avoided.

Deep dive

Constructor injection and setter injection differ mainly in when dependencies are supplied and whether the object can exist without them.

How it works
  • Constructor injection supplies required dependencies before the object becomes usable.
  • It supports final fields and immutable design.
  • It makes unit testing straightforward.
  • Setter injection is useful when a dependency is optional or may be replaced after construction.
Example / mental model
class OrderService {
    private final PaymentService paymentService;

    OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}
Common pitfalls
  • Too many constructor dependencies can reveal that the class has too many responsibilities.
  • Field injection hides required collaborators.
Remember: Required dependency? Prefer constructor injection. Truly optional dependency? Setter can make sense.
